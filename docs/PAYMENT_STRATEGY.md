# EnterLab 결제 전략 문서

> 작성: 2026-05-12
> 컨텍스트: 1:1 코칭/웹 개발 의뢰 (단가 100만~1,000만원) 한국 B2C/B2B 결제 시스템 설계
> 본 문서는 결제 관련 모든 조사·논의·결정을 누적 기록.

---

## 0. TL;DR

EnterLab은 **2개의 별개 서비스**가 있으며, 각각 결제 패턴이 다릅니다:

| 항목 | 코칭 (STARTER/STANDARD/PREMIUM) | 웹 개발 의뢰 (BASIC/STANDARD/PREMIUM) |
|------|----------|----------|
| **단가** | 98~249만원 (정가) | 290~1,000만원+ (견적, 변동) |
| **결제 패턴** | Pattern B (결제 링크, 1회) | Pattern B + 분납 (또는 Pattern C) |
| **결제 시점** | 코칭 시작 전 1회 | 계약금→중도금→잔금 (3단계) |
| **고객 유형** | B2C 위주 | **B2B 비중 ↑** (사업자) |
| **결제 수단** | 카드/카카오페이/무통장 | 무통장 (B2B) + 카드 (B2C) |
| **세금계산서** | 요청 시 발급 | **거의 필수** (B2B) |
| **할부** | 카드 무이자 2~6개월 | 분납 (계약/중도/잔) |
| **계약서** | 약관 동의로 갈음 | **별도 계약서 필수** |
| **공통** | PG: 토스페이먼츠, 부가세 정책, 환불 정책 |

### 공통 결정
- **PG**: 토스페이먼츠 (미가입 상태)
- **부가세**: 사업자 분류 확정 필요 (면세 가능성)
- **활성화 시점**: Q1 (2026년 5~7월) 사용자 결정 후
- **현 스캐폴드**: ✅ Payment 모델 + Billing adapter + /billing 라우트 (default off)

## 2. 한국 결제 시장 조사 (2026-05)

### 2.1 결제 패턴 4가지

#### Pattern A: 셀프 즉시 결제
- 사례: 클래스101, 야나두, 패스트캠퍼스, 콜로소
- 단가: 수만~수십만원
- 특징: 페이지 → 결제 → 영수증, 자동화 100%
- **EnterLab 적합도**: ❌ (단가/모델 부적합)

#### Pattern B: 무료 상담 → 결제 링크 ⭐
- 사례: 1:1 컨설팅, 고가 코칭, 디자인 외주, 변호사
- 단가: 100만~수천만원
- 특징: 사람 개입 후 결제 자동
- **EnterLab 적합도**: ✅ **권장**

#### Pattern C: 견적서 → 무통장 입금 (B2B)
- 사례: 기관/기업 영업
- 단가: 수백만~수억원
- 특징: 자동화 없음, 신뢰 기반
- **EnterLab 적합도**: ⚠️ B2B 출시 시 (Q4)

#### Pattern D: 정기 구독 결제
- 사례: 트레바리, 윌라, 멤버십
- 단가: 월/연 구독
- **EnterLab 적합도**: ❌ (일회성 코칭)

### 2.2 한국 결제 수단 점유율

| 수단 | 점유율 | EnterLab 적합 | 비고 |
|------|------|------------|------|
| 신용카드 | 60%+ | ⭐⭐⭐⭐⭐ | 무이자 할부 핵심 |
| 카카오페이 | 15% | ⭐⭐⭐⭐⭐ | 시니어/소상공인 친화 |
| 네이버페이 | 10% | ⭐⭐⭐⭐ | 네이버 이용자 |
| 무통장 입금 | 8% | ⭐⭐⭐⭐ | 시니어 일부 선호 |
| 토스페이 | 5% | ⭐⭐⭐ | 젊은 사용자 |
| 삼성/Apple Pay | 2% | ⭐⭐ | 모바일 |
| 휴대폰 결제 | <1% | ❌ | 100만 한도 |

### 2.3 비슷한 한국 코칭 사이트 사례

## 3. PG 비교 (토스페이먼츠 vs 포트원 vs 나이스페이)

| 항목 | 토스페이먼츠 ⭐ | 포트원 (아임포트) | 나이스페이 |
|------|---------------|-----------------|-----------|
| 수수료 | 2.8% + ₩200 | PG별 2.7~3% | 협상 |
| 가입 난이도 | 쉬움 (사업자 인증) | 쉬움 | 영업 협의 |
| API 문서 | 우수 | 우수 (PG 추상화) | 보통 |
| 빌링키 (정기결제) | ✅ | ✅ | ✅ |
| 세금계산서 옵션 | ✅ 자체 + Bill36500 | ✅ Bill36500 통합 | ✅ |
| 카카오페이 | ✅ | ✅ | ✅ |
| 네이버페이 | ✅ | ✅ | ✅ |
| Apple Pay | ✅ (2024~) | ✅ | △ |
| 결제 링크 발급 | ✅ Toss 자체 페이지 | ✅ 포트원 호스팅 | △ |
| Ruby/Rails SDK | 비공식 (REST API 직접) | 비공식 | 비공식 |
| 개발자 친화도 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |
| **EnterLab 권장** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |

→ **토스페이먼츠 권장**.

### 3.1 토스페이먼츠 결제 흐름

```
1. 회사: POST /v1/payments/{paymentKey}/check (결제 생성)
2. 사용자: 결제 페이지에서 카드/카카오페이 등 선택
3. 토스: 결제 인증 (3DS 등)
4. 회사: POST /v1/payments/confirm (승인)
5. 토스 webhook → 회사 (결제 결과)
6. 회사: Payment.mark_paid!
```

## 4. 서비스별 권장 아키텍처

### 4.1 코칭 (STARTER/STANDARD/PREMIUM) — Pattern B

**특징**: 정가, 1회 결제, B2C 위주, 단가 100~250만원

```
[1] 사이트: 사용자 → 무료 상담 신청
       ↓
[2] 코치: 카톡/전화로 1:1 상담
       ↓
[3] 합의: 패키지(STARTER/STANDARD/PREMIUM)·일정·금액 확정
       ↓
[4] 관리자 페이지: 코치가 "결제 링크 발송" 클릭
       ├── Payment 레코드 생성 (status=issued, amount=패키지 정가)
       ├── pg_order_id 생성 (예: "COACH-{reservation_id}-{random}")
       ├── 토스 API: 결제 페이지 URL 발급 (무이자 할부 옵션)
       └── 카톡 알림톡/SMS/이메일로 링크 전송
       ↓
[5] 수강생: 링크 클릭 → 토스 결제 페이지
       ├── 카드 (2~6개월 무이자 할부)
       ├── 카카오페이
       ├── 네이버페이
       └── 무통장 입금 (드뭄)
       ↓
[6] 토스: 결제 처리 → 자동 현금영수증
       ↓
[7] Webhook: 결과 회사로
       ├── Payment.mark_paid!
       ├── Reservation 결제 완료 표시
       └── 카톡 알림톡 (수강생 + 관리자)
       ↓
[8] 코칭 시작 → 마이페이지 진척률 표시
```

**Payment 레코드 수**: **1개** (한 번에 전액)

### 4.2 웹 개발 의뢰 (BASIC/STANDARD/PREMIUM) — Pattern B + 분납

**특징**: 견적 기반, 분할 결제, B2B 비중 ↑, 단가 290만~1,000만원+

```
[1] 사이트: 의뢰인 → 무료 상담 신청 (BASIC/STANDARD/PREMIUM 표시)
       ↓
[2] 코치: 카톡/전화로 요구사항 청취 + PRD 초안
       ↓
[3] 견적서 작성:
       ├── 작업 범위 명세 (PRD)
       ├── 단계별 산출물
       ├── 최종 견적 금액 (정가 290만 ~ 협의)
       └── 결제 일정 (3단계 분납)
       ↓
[4] 계약서 작성/서명 (별도 PDF)
       ├── 작업 범위
       ├── 결제 일정
       ├── 저작권 (산출물 의뢰인 귀속)
       └── 분쟁 해결
       ↓
[5] 관리자 페이지: "분납 결제 링크 3건 발송"
       ├── Payment 1 (계약금 30%, status=issued)
       ├── Payment 2 (중도금 40%, status=scheduled — 중간 단계 시 발송)
       └── Payment 3 (잔금 30%, status=scheduled — 납품 시 발송)
       ↓
[6] 의뢰인: 계약금 결제
       ├── B2C: 카드 / 카카오페이
       └── B2B: 무통장 + 세금계산서 (사업자번호 입력)
       ↓
[7] 작업 진행 → 마일스톤 도달
       ↓ (관리자: "중도금 결제 링크 발송" 클릭)
[8] 중도금 결제
       ↓
[9] 작업 완료 → 검수 → 납품
       ↓ (관리자: "잔금 결제 링크 발송" 클릭)
[10] 잔금 결제 → 세금계산서 최종 발행
```

**Payment 레코드 수**: **3개** (계약금/중도금/잔금)
**대안**: 견적이 작거나 의뢰인이 일시불 원하면 1회 결제도 가능

### 4.3 결제 데이터 모델 보강 (현 스캐폴드 + 추가)

```ruby
# payments 테이블 (현재 + 추가 필드 검토)
- reservation_id
- amount (원)
- status: pending | issued | scheduled | paid | failed | refunded | partially_refunded
- pg_provider: toss | kakaopay | naverpay
- pg_payment_key (PG 발급)
- pg_order_id (회사 발급)
- pg_method: 카드 | 계좌이체 | 간편결제
- issued_at, paid_at, refunded_at
- refunded_amount
- memo (관리자)

# 향후 추가 검토 (분납 지원 시):
- payment_phase: full | deposit | middle | final  # 계약금/중도금/잔금
- phase_order: 1 / 2 / 3  # 결제 순서
- requires_tax_invoice: boolean  # 세금계산서 발급 요청
- business_registration_no: string  # 사업자번호 (B2B)
- contract_id: reference  # 계약서 ID (B2B 웹개발)
```

→ **현 스캐폴드는 단순 1회 결제 가정**. 분납/B2B 지원 시 스키마 확장 필요.

### 4.4 service_type별 라우팅

`Reservation.service_type`이 이미 있음 (`coaching` / `app_development`).
→ 결제 흐름도 service_type에 따라 분기 가능:

```ruby
# 의사 코드
case reservation.service_type
when "coaching"
  CoachingBillingFlow.new(reservation).issue_link  # 1회 결제
when "app_development"
  WebDevBillingFlow.new(reservation).issue_phased_links  # 3단계 분납
end
```

## 6. 무이자 할부 전략

코칭 단가 100~250만원 → 일시불 부담 ↑ → **할부가 핵심 마케팅**

| 옵션 | 회사 비용 | 사용자 부담 |
|------|---------|----------|
| **카드사 무이자 할부** (2~6개월) | 0 | 0 (카드사 부담) |
| **가맹점 무이자 할부** | 약 1~2% 추가 수수료 | 0 |
| **분납** (1차 30% + 2차 70%) | 0 | 수동 관리 |

## 11. 웹 개발 의뢰 결제 상세 (B2B 포함)

웹 개발 의뢰는 코칭과 결제 흐름이 크게 다름.

### 11.1 결제 구조: 3단계 분납이 표준

한국 웹 개발 외주 업계 표준:

| 단계 | 비율 | 시점 | 비고 |
|------|------|------|------|
| **계약금** | 30% | 계약 체결 시 | 작업 착수 조건 |
| **중도금** | 40% | 중간 마일스톤 (보통 50%) | 진척 확인 후 |
| **잔금** | 30% | 검수 완료 후 | 납품/이관 후 |

**대안**:
- 일시불 (소액 의뢰 또는 의뢰인 요청 시): 100%
- 2단계 (50/50): 착수 + 완료
- 5단계 이상: 대규모 프로젝트만

### 11.2 B2C vs B2B 차이

| 항목 | B2C (개인 의뢰인) | B2B (사업자/기관) |
|------|----------------|--------------|
| 빈도 | 낮음 | **높음** (웹 개발의 70%+) |
| 결제 수단 | 카드/카카오페이 | **무통장 입금** 위주 |
| 계약서 | 간단 (이메일) | **별도 PDF 계약서 서명** |
| 세금계산서 | 요청 시 | **필수 발급** |
| 사업자번호 | 불요 | **필수 입력** |
| 결제 일정 | 빠른 (수일) | 느린 (자금집행 절차, 수주~수개월) |
| 분쟁 | 적음 | 중간 (관할 법원 합의 필요) |

### 11.3 B2B 흐름 (Pattern C 변형)

```
1) 의뢰 문의 (이메일/전화/사이트)
2) 1차 미팅 → 요구사항 청취
3) 견적서 PDF (사업자번호, 부가세 명시)
4) 계약서 작성:
   ├── 작업 범위 (PRD)
   ├── 산출물 명세
   ├── 분납 일정 (계약금 30% / 중도금 40% / 잔금 30%)
   ├── 저작권 (의뢰인 귀속)
   ├── 변경 사항 처리 (추가 비용)
   └── 분쟁 해결 조항
5) 양 당사자 서명 (전자/실물)
6) 사업자 정보 등록 (관리자 페이지)
7) 계약금 → 세금계산서 1차 발행 → 무통장 입금 → 수동 확인
8) 작업 진행 → 마일스톤 도달
9) 중도금 → 세금계산서 2차 → 무통장 입금
10) 검수 → 잔금 → 세금계산서 3차 → 납품
```

### 11.4 결제 자동화 vs 수동 운영

| 영역 | B2C (개인) | B2B (사업자) |
|------|-----------|------------|
| 결제 링크 발송 | 자동 (토스 결제 페이지) | 무통장 안내 (계좌번호) |
| 입금 확인 | 자동 (PG webhook) | 수동 (CMS, 은행 앱) |
| 세금계산서 | 요청 시 발급 | 자동 (Bill36500) |
| 영수증 | 자동 (토스) | 세금계산서로 갈음 |
| 환불 | 자동 (60일 이내) | 수동 (계좌이체) |

### 11.5 시스템 요구사항 (웹 개발 결제 활성화 시)

#### 필수 구현
- [ ] 견적서 PDF 자동 생성 (이름/주소/사업자번호/금액/부가세)
- [ ] 계약서 PDF 자동 생성 (작업 범위, 분납 일정)
- [ ] 분납 결제 관리 (Payment 레코드 3개 한 묶음)
- [ ] 사업자번호 입력/저장 (B2B)
- [ ] 세금계산서 자동 발급 (Bill36500 또는 토스 옵션)
- [ ] 입금 확인 워크플로 (B2B 무통장)
- [ ] 진척률 표시 (의뢰인 관리 페이지)

#### 선택 구현
- [ ] 오픈뱅킹 API (자동 입금 확인) — 비용 ↑
- [ ] 전자계약 (모두싸인, 클라우드사인) — 월 수만원
- [ ] PMS 통합 (작업 단계별 결제 트리거)

## 12. 분할 결제 / 분납

### 12.1 코칭 — 카드 무이자 할부 (간단)

| 옵션 | 처리 |
|------|------|
| **카드 무이자 할부** (2~6개월) | PG 자동 처리 |
| **2회 분납** | Payment 2개 (드뭄) |

→ 코칭은 카드 무이자 할부만으로 충분. 분납 거의 X.

### 12.2 웹 개발 — 3단계 분납 (표준)

| 단계 | Payment.payment_phase | 발송 시점 |
|------|---------------------|---------|
| 계약금 | deposit | 계약 체결 시 |
| 중도금 | middle | 작업 50% 진행 시 |
| 잔금 | final | 검수 완료 시 |

#### 구현 방식 (향후)

```ruby
# 의사 코드 — 분납 자동 생성
class WebDevPaymentFlow
  def initialize(reservation, total_amount)
    @reservation = reservation
    @total = total_amount
  end

  def issue_phased_payments
    deposit  = (@total * 0.3).to_i
    middle   = (@total * 0.4).to_i
    final    = @total - deposit - middle  # 반올림 보정

    Payment.create!(reservation: @reservation, amount: deposit,
                    status: "issued", payment_phase: "deposit", phase_order: 1)
    Payment.create!(reservation: @reservation, amount: middle,
                    status: "scheduled", payment_phase: "middle", phase_order: 2)
    Payment.create!(reservation: @reservation, amount: final,
                    status: "scheduled", payment_phase: "final", phase_order: 3)
  end
end
```

→ 현 스키마에 `payment_phase`, `phase_order` 필드 추가 필요 (분납 활성화 시 마이그레이션).

### 12.3 분납 결제 관리 UI (관리자)

```
예약 상세 페이지
├── 패키지: STANDARD 웹 개발 의뢰 (₩5,000,000)
├── 분납 결제 현황:
│   ├── 1차 계약금 ₩1,500,000 [✅ 완료] (2026-05-15)
│   ├── 2차 중도금 ₩2,000,000 [⏸ 대기]    [결제 링크 발송]
│   └── 3차 잔금 ₩1,500,000 [⏸ 대기]
└── 세금계산서 발행 현황: 1차 ✅ / 2차 ⏸ / 3차 ⏸
```
